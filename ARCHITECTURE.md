# Memory Insight — Architecture & Build Plan

> Generated by /office-hours on 2026-03-29 | Status: APPROVED

---

## Problem Statement

Memory forensics tooling is entirely enterprise-focused — expensive licenses, complex
installs, sales-led distribution. The sysadmin or DevOps engineer running 50 Linux
servers who wants scheduled memory surveillance has no tool. Nobody serves them.

Memory Insight closes that gap: lightweight Go daemon + cloud analysis pipeline +
web UI, distributed via curl install with an API key gated through a signup flow.

**Phase 1** ships the full pipeline and UI: scheduled dumps, Volatility3 analysis,
structured findings with severity classification and AI summaries. What Phase 1 does
NOT include: diff alerting (process appeared/disappeared since last scan). That is the
headline surveillance feature — it ships in Phase 2. Phase 1 is the pipe that makes
Phase 2 possible. Every Phase 1 scan generates a snapshot. Phase 2 diffs them.

---

## What Makes This Cool

- **One-liner install:** `curl -sSL https://get.memoryinsight.io | bash -s -- --key sk_live_xxxx --namespace ns_xxxxxxxx`
- **Scheduled surveillance:** Set a cron, forget it. Phase 1: findings per scan. Phase 2: diff alerts when memory changes.
- **Two audiences, one tool:** Raw output (expandable) for security researchers who
  want hex dumps. Plain-English AI summaries for sysadmins who just want "is this bad?"
- **No inbound ports needed:** Agent long-polls outbound only. Works through any NAT,
  firewall, or cloud security group.

---

## Constraints

- Open-core project — agent and worker are open source; hosted backend (API + UI)
  requires signup. Self-hosting the full stack is a valid use case documented separately.
- Volatility3 analysis must run offline in isolated containers (hostile input risk).
  Network isolation is enforced post-startup via seccomp/network namespace: ECR image
  pull and initial S3 dump download are permitted; all subsequent outbound traffic is blocked.
- Agent must be a single static Go binary — no runtime dependencies on user machine.
- Free tier must include scheduled scans — surveillance is the hook, not the upsell.

---

## Premises

1. **Core value is surveillance, not incident response.** The product earns its keep
   between incidents. Diff alerts on scheduled scans are the differentiator.

2. **End-to-end pipe first is correct.** The Volatility3 worker is the hardest piece.
   Build the pipe, then layer intelligence on top.

3. **Agent = single static Go binary via curl.** Any install friction kills adoption
   in the target segment (sysadmins who don't have a security team to configure things).

4. **Scheduled scans must be in the free tier.** Gating surveillance behind a paywall
   defeats the product's core purpose and kills the "oh no" moment.

5. **Volatility3 analysis runs with post-startup network isolation.** Workers analyzing
   potentially hostile memory dumps block all outbound network access after ECR image
   pull and S3 dump download complete. Implemented via seccomp profile or network
   namespace — not by removing network at the Fargate task level (architecturally
   impossible since ECR pull requires network).

## Architecture Decisions

### Auth Architecture
**One auth system for everything: Better Auth.**

Better Auth runs as a shared instance initialized with the same `DATABASE_URL` and
`BETTER_AUTH_SECRET` in both Next.js and Fastify. No session bridging, no shared-secret
JWT, no custom key table — all auth state lives in Better Auth's Postgres tables.

**Plugins used:**

| Plugin | What it handles |
|--------|----------------|
| `emailPassword` | Sign up / sign in with email + password |
| `socialProvider("github")` | GitHub OAuth |
| `organization` | Org creation, invitations, membership, role assignment |
| `apiKey` | Key generation (sha256 stored), validation, per-key rate limiting, expiry |
| `access` | Fine-grained RBAC — maps org roles (admin/write/read) to permissions |

**How each service uses Better Auth:**

- **Next.js**: standard Better Auth handler (`auth.handler`) — manages sessions, OAuth
  callbacks, org management UI routes, API key creation UI. Better Auth owns the
  `user`, `session`, `account`, `organization`, `member`, `invitation`, `apikey` tables.

- **Fastify**: initializes the same Better Auth instance (same env vars, same DB).
  Two middleware functions, one per credential type:
  - `sessionAuth`: `await auth.api.getSession({ headers: req.headers })` → resolves user + org membership + role
  - `apiKeyAuth`: `await auth.api.verifyApiKey({ key: bearerToken })` → resolves org_id + key metadata

Agent sends `Authorization: Bearer sk_live_xxxx`. Fastify's `apiKeyAuth` middleware calls
Better Auth to validate. No custom hashing code, no custom key table.

```
Browser ──session cookie──▶ Next.js (Better Auth handler)
                                  │  org mgmt, key creation
                                  ▼
                            Better Auth ──▶ Postgres (Better Auth tables)
                                  ▲
                                  │  verifyApiKey / getSession
Agent ──Bearer sk_live_xxxx──▶ Fastify (Better Auth instance, same DB)
```

**RBAC via Better Auth access plugin:**
Org roles (`admin` / `write` / `read`) are stored in Better Auth's `member` table.
The `access` plugin maps roles to permissions (e.g., `machine:trigger`, `namespace:create`).
Fastify middleware resolves role from Better Auth, then checks the required permission
for the route. Zero custom RBAC code.

### Queue Architecture
Two distinct queues, two distinct purposes:

- **BullMQ + Redis (scheduling + long-poll wakeup path):** BullMQ handles repeating
  schedules (cron). When a scheduled job fires, BullMQ creates a `jobs` record and
  publishes to Redis `machine:{id}:jobs`. The long-poll handler subscribes to this
  channel when an agent connects and wakes the response on publish. Manual triggers
  (from UI) skip BullMQ and publish directly to Redis. Redis pub/sub is the wakeup
  mechanism for long-poll across all Fastify instances.

- **SQS (API → Lambda → ECS worker path):** When an agent completes upload and
  calls `/jobs/:id/complete`, the API posts a message to SQS. Lambda dispatcher reads
  SQS, inspects dump size, launches an appropriately-sized ECS Fargate task.
  These are independent queues. BullMQ is not involved in the analysis dispatch path.

### Volatility3 Symbol Tables (ISF Files)
**Decision: Bundle in Docker image. CDN pull is not permitted (violates post-startup isolation).**

Phase 1 coverage matrix (bundled in worker Docker image):
- Ubuntu 22.04 LTS (kernel 5.15.x) — primary target
- Ubuntu 20.04 LTS (kernel 5.4.x)
- Debian 12 (kernel 6.1.x)
- Amazon Linux 2023 (kernel 6.1.x)
- RHEL 9 / Rocky Linux 9 (kernel 5.14.x)

ISF files sourced from: https://github.com/Abyss-W4tcher/volatility3-symbols
Worker will log "unsupported kernel" and write a FAILED finding if no matching ISF
is found. Alpine Linux (musl libc) is explicitly out of scope for Phase 1 — AVML
does not support musl.

### Scheduled Job Ownership
**Decision: Server-side, owned by the API.**

The API creates BullMQ repeating jobs using `cron_expr` from the `schedules` table.
On startup, the API reads all enabled schedules and registers them with BullMQ.
When a repeating job fires, BullMQ creates a `jobs` record and makes it available
to the polling agent. Agent requires no changes to support scheduling — it just
receives another `PENDING` job.

### Worker Dispatcher Design

```
Agent: POST /jobs/:id/complete { s3_key, size_bytes, checksum }
        │
        ▼
Fastify API
    validates: job.machine.org_id matches calling API key's org_id (ownership check)
    publishes: SQS message { job_id, s3_key, size_bytes }
        │
        ▼
SQS queue (standard, 1 consumer)
        │
        ▼
Lambda dispatcher (Python, ~50ms)
    reads: job_id, s3_key, size_bytes from SQS message
    decision:
        < 4 GB  → ECS RunTask(taskDef: worker-small,  vCPU: 2, mem: 8192)
        4-16 GB → ECS RunTask(taskDef: worker-medium, vCPU: 4, mem: 16384)
        > 16 GB → ECS RunTask(taskDef: worker-large,  vCPU: 8, mem: 65536)
    writes: analyses record (status: running, worker_size)
    passes: job_id, s3_key, analysis_id as environment variables to Fargate task
```

Fargate task networking (VPC isolation model):
- VPC with NAT gateway for outbound internet (worker needs DB write + Claude API)
- ECR VPC endpoint: Fargate pulls worker image without touching internet
- S3 VPC endpoint: dump download goes through VPC, zero internet transfer cost
- Security group: allows outbound HTTPS (443) only — blocks all other outbound ports
- Volatility3 subprocess isolation: Python's `subprocess.run()` with `network_namespaces`
  via `unshare --net` (Linux). The subprocess that runs Volatility3 plugins has no
  network access. The parent Python process retains HTTPS access for DB and Claude API.

**Claude API key management:** stored in AWS SSM Parameter Store (`SecureString`).
Fargate task role has `ssm:GetParameter` permission for the specific parameter ARN.
Worker reads key at startup via `boto3.client('ssm').get_parameter()`. Never in env vars,
never in Docker image, never in source code.

### AVML Embedding in Agent Binary
AVML binaries embedded using `//go:embed avml/avml-linux-amd64 avml/avml-linux-arm64`.
At runtime, the agent writes the correct arch binary to `/tmp/avml-{version}`, sets
executable bit, runs it, then removes it. Binary size impact: ~25MB per arch, ~50MB
total added to the Go binary. Acceptable for a daemon that installs once.

### Deployment Target
**Fly.io for API + Redis. AWS S3 for object storage.**

R2 does not natively emit events to AWS SQS (only webhook HTTP callbacks). S3 integrates
natively with SQS and Lambda. Given workers run on AWS Fargate anyway, S3 is the correct
storage choice.

- API + Redis + Postgres: Fly.io (free tiers, simple deploys)
- Object storage: AWS S3 (us-east-1)
- Lambda dispatcher + ECS Fargate: AWS us-east-1
- Agent pulls presigned PUT URLs from API; uploads directly to S3

**Worker Docker image: ECR for Fargate, GHCR for local dev.**
GHCR requires internet access to pull — no VPC endpoint equivalent. Fargate uses ECR via
VPC endpoint (no internet). GitHub Actions pushes to both on merge to main:
- `ghcr.io/memoryinsight/worker:latest` — local development
- `<account>.dkr.ecr.us-east-1.amazonaws.com/memoryinsight-worker:latest` — Fargate

**Analysis dispatch: API-driven, not S3 ObjectCreated.**
Agent calls `POST /jobs/:id/complete` → API publishes to SQS → Lambda dispatches Fargate.
Eliminates S3 event setup complexity. Agent calling complete is the authoritative signal.

### Finding Severity Classification
Rule-based for Phase 1. ML/AI scoring is Phase 3.

| Plugin | Default | CRITICAL condition | HIGH condition |
|--------|---------|-------------------|----------------|
| malfind | HIGH | Executable pages + no backing file on disk | Any RWX region |
| pslist | INFO | Hollow process (VAD mapped, no backing file) | PID mismatch with pstree |
| netscan | MEDIUM | Connection to known-bad IP (threat intel, Phase 3) | Outbound to non-RFC1918 on unusual port |
| cmdline | LOW | Shell spawned from non-shell parent | Process with base64-encoded args |
| linux.proc.Maps | MEDIUM | Shared library loaded from /tmp or /home path | Anonymous executable mapping |
| envars | LOW | Contains "KEY", "SECRET", "PASSWORD", "TOKEN" | — |

### Agent Failure Path
```
Agent error conditions → POST /jobs/:id/fail  { error_code, message }

Error codes:
  AVML_FAILED         — AVML returned non-zero exit code
  UPLOAD_FAILED       — S3 multipart upload failed after 3 retries
  DISK_FULL           — insufficient /tmp space (check before dump, need 2× RAM)
  PERMISSION_DENIED   — not running as root / missing CAP_SYS_RAWIO

Retry policy:
  AVML_FAILED:    no retry (kernel issue, not transient)
  UPLOAD_FAILED:  retry 3× with exponential backoff (5s, 25s, 125s)
  DISK_FULL:      no retry (fail fast, alert user)
  PERMISSION:     no retry (configuration problem)

UI: job status timeline shows FAILED state with error_code + message visible to user.
```

### API Failure Paths

**Redis connection lost during long-poll:**
```
Long-poll handler opens Redis SUBSCRIBE for "machine:{id}:jobs"
  → if Redis errors mid-subscription:
      resolve the pending Fastify reply with 204 immediately
      close the Redis subscriber client
      log the error (don't crash the Fastify instance)
  → agent receives 204, re-polls immediately
  → next poll attempt will fail at SUBSCRIBE if Redis is still down → 503 to agent
  → agent backs off (exponential, 5s/25s/125s) and retries
```

**SQS publish failure on job completion:**
```
Agent: POST /jobs/:id/complete { s3_key, size_bytes, checksum }
  → Fastify updates job status → QUEUED
  → SQS publish attempt fails:
      set job.status = FAILED
      set job.error_code = 'SQS_PUBLISH_FAILED'
      set job.error_message = <SQS error message>
      return 500 to agent (agent does NOT retry /complete — it calls /fail instead)
  → UI shows FAILED with error visible
  → operator can manually re-trigger from UI (creates a new job)
```

### Performance SLA
15-minute target applies to same-region uploads (EC2 agent in us-east-1, S3 in us-east-1):
- AVML dump: ~60-90s for 4GB RAM
- S3 upload (same-region, ~1Gbps EC2): ~35s
- Lambda dispatch: <1s
- Fargate cold start: ~30-60s
- Volatility3 analysis (4 plugins on 4GB): ~4-6 min on 2 vCPU
- DB write + AI summaries: ~30s
- **Total: ~8-10 min. 15-min SLA achievable same-region.**

Cross-region or residential connections: SLA not guaranteed (4GB at 50Mbps = ~10.7 min upload alone).

These are handled automatically by Better Auth's organization and apiKey plugins.
No custom Fastify routes needed. Better Auth mounts them under `/api/auth/*` in Next.js.

|--------|------|------|-------------|

| Method | Path | Auth | Description |
|--------|------|------|-------------|

| GET | `/machines` | Session (Read+) | List org's machines |

| DELETE | `/machines/:id` | Session (Write+) | Deregister machine |
| GET | `/jobs/pending` | API Key | Long-poll (30s hang, 204 on timeout, 200+job on match) |
| POST | `/jobs` | Session (Write+) | Trigger manual job |
| POST | `/jobs/:id/complete` | API Key | Mark dump uploaded → publishes to SQS |
| POST | `/jobs/:id/fail` | API Key | Record failure with error_code |
| GET | `/dumps/presigned-url` | API Key | `?job_id=&size_bytes=` → `{ upload_url, s3_key }` |
| GET | `/results/:job_id` | Session (Read+) | Full analysis results |
| GET | `/findings/:job_id` | Session (Read+) | Findings with severity |
| POST | `/schedules` | Session (Write+) | Set cron schedule for a machine |
| GET | `/schedules/:machine_id` | Session (Read+) | Get machine's schedule |

Authorization rules:
- **Session endpoints:** Fastify calls `auth.api.getSession()` → resolves user ID → looks up org membership + role in Better Auth's `member` table → checks required permission via `access` plugin.
- **API key endpoints:** Fastify calls `auth.api.verifyApiKey()` → returns `org_id` + key metadata. Ownership check: `job.machine.org_id === key.org_id`.
- Insufficient role → 403. Resource in wrong org → 403. Invalid key → 401.

Long-poll behavior: 30s hang, 204 on timeout, agent re-polls immediately.
Redis pub/sub wakeup — works across multiple Fastify instances, no listener leak.
One PENDING job per machine at a time (BullMQ duplicate guard on key `job:{machine_id}`).
Schedules deduplicated on key `schedule:{schedule_id}`. New schedules register
immediately (no API restart needed). 24h TTL on unclaimed jobs.

## Organization & Namespace Hierarchy

```
Organization
└── Namespace
    └── Machines
```

Every resource is scoped to an org. Namespaces are logical groupings within an org
(e.g. `production`, `staging`, `eu-west`). Machines belong to a namespace.

### Onboarding Flow
1. User signs up via Better Auth
2. Prompted to: **Create an organization** (becomes Admin) or **Request to join** an existing one (pending until an Admin approves)
3. Admin creates one or more **Namespaces** within the org
4. Admin creates an **API Key** scoped to the org
5. User runs install script with both the API key and a namespace ID:

```bash
curl -sSL https://get.memoryinsight.io | bash -s -- \
  --key sk_live_xxxx \
  --namespace ns_xxxxxxxx
```

The API key resolves the org. The namespace ID resolves where the machine is placed within it.

### RBAC

Roles and permissions are declared in Better Auth's `access` plugin config. Three org roles:

| Permission | Admin | Write | Read |
|------------|-------|-------|------|
| Invite / approve members | ✓ | — | — |
| Remove members | ✓ | — | — |
| Create / delete API keys | ✓ | — | — |
| Create / delete namespaces | ✓ | — | — |
| Trigger dumps (manual jobs) | ✓ | ✓ | — |
| Edit schedules | ✓ | ✓ | — |
| View machines & job history | ✓ | ✓ | ✓ |
| View findings & analysis results | ✓ | ✓ | ✓ |
| View raw dump logs | ✓ | ✓ | ✓ |
| Download dumps (if retained) | ✓ | ✓ | — |
| Deregister machines | ✓ | ✓ | — |

Permission check in Fastify: `auth.api.hasPermission({ userId, orgId, permission: "machine:trigger" })`.
Better Auth enforces it — no custom middleware table logic.

Better Auth's `organization` plugin defaults to owner/admin/member roles. We configure
custom role names (admin/write/read) and map the permission matrix above via the `access` plugin.

## Database Schema

Better Auth owns its own tables (`user`, `session`, `account`, `verification`,
`organization`, `member`, `invitation`, `apikey`) — these are created automatically
by `better-auth generate` and are not duplicated here.

Our schema only contains application-domain tables. All foreign keys that reference
users or orgs use the string IDs that Better Auth generates.

```sql
-- Namespaces (logical groups within a Better Auth org)
-- org_id references Better Auth's `organization.id` (TEXT, not UUID)
CREATE TABLE namespaces (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      TEXT NOT NULL,           -- Better Auth organization.id
  name        TEXT NOT NULL,
  slug        TEXT NOT NULL,           -- e.g. "production", "staging"
  created_at  TIMESTAMPTZ DEFAULT now(),
  UNIQUE(org_id, slug)
);

-- Indexes (explicit, non-obvious ones)
-- Ownership check on job endpoints: job → machine → org_id
CREATE INDEX idx_machines_id_org ON machines(id, org_id);
-- Long-poll lookup by machine_id
CREATE INDEX idx_jobs_machine_status ON jobs(machine_id, status) WHERE status = 'PENDING';

-- Machines (belong to a namespace, scoped to an org)
CREATE TABLE machines (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  namespace_id    UUID REFERENCES namespaces(id) ON DELETE RESTRICT,
  -- ON DELETE RESTRICT: deleting a namespace requires deregistering its machines first
  org_id          TEXT NOT NULL,       -- Better Auth organization.id (denormalized for fast ownership checks)
  hostname        TEXT NOT NULL,
  os              TEXT NOT NULL DEFAULT 'linux',
  distro          TEXT,
  arch            TEXT,
  total_memory_gb INTEGER,
  agent_version   TEXT,
  last_seen       TIMESTAMPTZ,
  status          TEXT DEFAULT 'active',  -- active | inactive | deregistered
  created_at      TIMESTAMPTZ DEFAULT now(),
  UNIQUE(namespace_id, hostname)       -- one record per hostname per namespace
);

-- Jobs (dump requests)
CREATE TABLE jobs (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  machine_id      UUID REFERENCES machines(id),
  triggered_by    TEXT,               -- Better Auth user.id, null if scheduled
  trigger_type    TEXT NOT NULL,      -- manual | scheduled
  status          TEXT NOT NULL DEFAULT 'PENDING',
  -- PENDING → DUMPING → UPLOADING → QUEUED → ANALYZING → DONE | FAILED
  scheduled_at    TIMESTAMPTZ,
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  error_code      TEXT,
  error_message   TEXT,
  created_at      TIMESTAMPTZ DEFAULT now()
);

-- Dumps (uploaded memory files)
CREATE TABLE dumps (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_id        UUID REFERENCES jobs(id),
  s3_key        TEXT NOT NULL,
  size_bytes    BIGINT,
  checksum      TEXT,
  uploaded_at   TIMESTAMPTZ DEFAULT now()
);

-- Analyses (worker run metadata)
CREATE TABLE analyses (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dump_id         UUID REFERENCES dumps(id),
  worker_size     TEXT,               -- small | medium | large
  plugins_run     TEXT[],
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  status          TEXT DEFAULT 'running'  -- running | done | failed
);

-- Findings (individual plugin results)
CREATE TABLE findings (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  analysis_id   UUID REFERENCES analyses(id),
  plugin        TEXT NOT NULL,
  severity      TEXT NOT NULL,        -- CRITICAL | HIGH | MEDIUM | LOW | INFO
  title         TEXT NOT NULL,
  raw_output    JSONB,
  ai_summary    TEXT,                 -- null if outside top-10 findings for job
  created_at    TIMESTAMPTZ DEFAULT now()
);

-- Schedules (per machine, managed by Write+ role)
CREATE TABLE schedules (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  machine_id  UUID REFERENCES machines(id) ON DELETE CASCADE,
  created_by  TEXT,                   -- Better Auth user.id
  cron_expr   TEXT NOT NULL,
  enabled     BOOLEAN DEFAULT true,
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

## Agent Flow

### Install
```bash
curl -sSL https://get.memoryinsight.io | bash -s -- \
  --key sk_live_xxxx \
  --namespace ns_xxxxxxxx
```
1. Detect arch (amd64 / arm64)
2. Download `meminsight-agent-linux-{arch}` from GitHub Releases
3. Call `meminsight-agent register` → `POST /machines/register` (org API key used once)
4. Receive `{ machine_id, machine_token }` from API
5. Write `/etc/meminsight/config.yaml` with `machine_id`, `machine_token`, `api_url`
   — **org API key is not written to disk**
6. Install + start `meminsight-agent.service` (systemd, runs as root)

The org API key (`sk_live_xxxx`) authenticates registration only. The API returns a
per-machine token (`mt_xxxx`) scoped to that machine_id. All subsequent daemon calls
use the machine token. Revoking a compromised machine token does not affect the org key
or any other machine.

Machine tokens are Better Auth `apiKey` records with `prefix: "mt_"` and
`metadata: { machine_id }`. Fastify validates via `auth.api.verifyApiKey()` and
confirms `metadata.machine_id` matches the route context.

### Registration Payload
```
POST /machines/register
Authorization: Bearer sk_live_xxxx    ← org API key, used ONCE

Body:
{
  "namespace_id": "ns_xxxxxxxx",
  "hostname": "prod-server-01",
  "os": "linux",
  "distro": "Ubuntu 22.04",
  "arch": "amd64",
  "total_memory_gb": 32,
  "agent_version": "0.1.0"
}

Response:
{
  "machine_id": "mch_xxxxxxxx",
  "machine_token": "mt_xxxxxxxxxxxx"  ← stored in config.yaml, never shown again
}
```

### Job Loop (long-poll with Redis pub/sub)
```
Agent  →  GET /jobs/pending            (Authorization: Bearer mt_xxxx, hangs up to 30s)
Fastify:  validate machine token
          → SUBSCRIBE Redis "machine:{id}:jobs"
          → wait (up to 30s)

On job created (UI trigger or BullMQ schedule fires):
  Fastify creates jobs record
  → Redis PUBLISH "machine:{id}:jobs" { job_id }
  → waiting connection wakes → 200 { job_id, type: "dump" }
  → Fastify UNSUBSCRIBEs channel

On timeout (30s with no job):
  → 204 No Content
  → agent re-polls immediately

Result: works correctly across multiple Fastify instances.
        No listener leak — UNSUBSCRIBE on resolve or timeout.
```

### Dump + Upload Flow
```
1. Agent receives job
2. Pre-flight: verify /tmp has ≥ 2× RAM free space
3. Extract AVML binary from embedded bytes → /tmp/avml-{version}
4. Run: avml /tmp/meminsight-{job_id}.lime
5. Call: GET /dumps/presigned-url?job_id=xxx&size_bytes=yyy
6. Receive: { upload_url, s3_key }  (expires 15 min)
7. Multipart PUT to S3 (64MB chunks, 3-retry with backoff on failure)
8. On success: POST /jobs/{job_id}/complete { s3_key, size_bytes, checksum }
9. On failure: POST /jobs/{job_id}/fail { error_code, message }
10. Clean up /tmp/meminsight-{job_id}.lime
```

API never touches dump bytes — S3 upload is direct from agent.

## Analysis Worker Flow

### Trigger
Agent `POST /jobs/:id/complete` → API publishes to SQS → Lambda reads SQS → ECS RunTask

### Kernel Detection
```bash
strings /tmp/dump.lime | grep -E "Linux version [0-9]" | head -1
```
Match against bundled ISF files. Write FAILED analysis record and exit if no match.

### Worker Steps
1. Read `JOB_ID`, `S3_KEY`, `ANALYSIS_ID` from environment
2. Fetch Claude API key from SSM Parameter Store at startup
3. Download dump from S3 via VPC endpoint → `/tmp/dump.lime`
4. Detect kernel version, match ISF file
5. For each plugin, run Volatility3 via `subprocess.run()` with `unshare --net`
   (subprocess has no network; parent process retains HTTPS for DB + Claude API)
6. Apply severity classification rules
7. Write findings to DB (plugin, severity, raw_output JSONB)
8. For top 10 findings by severity: call Claude API for plain-English summary
   (~150 token prompt + raw excerpt → ~100 token output, ~$0.015/job max)
9. Update analysis record: status=done, completed_at

### Worker Sizing
| Dump Size | Task Definition | vCPU | RAM | Ephemeral Storage | Est. Duration |
|-----------|----------------|------|-----|-------------------|---------------|
| < 4 GB | worker-small | 2 | 8 GB | 32 GB | ~5 min |
| 4–16 GB | worker-medium | 4 | 16 GB | 64 GB | ~15 min |
| > 16 GB | worker-large | 8 | 64 GB | 200 GB | ~40 min |

`ephemeralStorageGiB` must be set explicitly in each ECS task definition. Fargate
default is 20GB — silently insufficient for any dump larger than ~14GB after accounting
for OS, image, and ISF files. Storage cost: ~$0.000111/GB/hour (adds ~$1.48 for a
200GB large worker running 40 min).

## Security

### Agent
- Outbound connections only — no listening port
- Machine token (`mt_xxxx`) scoped to one machine — compromise of one box doesn't expose other machines or the org API key
- Org API key used only at registration, never stored on agent machines
- Dumps encrypted at rest in S3 (AES-256, SSE-S3)
- Presigned URLs expire in 15 minutes
- TLS everywhere

### Worker
- Fresh container per analysis (no state reuse)
- Volatility3 subprocess runs in network namespace (`unshare --net`) — no outbound
- Parent process: HTTPS only (DB write, Claude API via SSM key)
- Memory + CPU limits enforced by ECS task definition
- Claude API key in SSM Parameter Store — never in env vars or image

### API
- Better Auth handles all of: session auth, API key auth, org membership, RBAC — zero custom auth code
- API keys: Better Auth `apiKey` plugin stores SHA-256 hash, returns plaintext once, supports per-key rate limiting
- Three-layer authorization enforced by Better Auth + Fastify middleware:
  1. Valid session (`auth.api.getSession`) or valid API key (`auth.api.verifyApiKey`)
  2. Resource `org_id` matches the resolved org from the credential
  3. Caller's role has the required permission (`auth.api.hasPermission`)
- API key create/delete: Admin role only (enforced via Better Auth access plugin)
- Agent endpoints accept Bearer API keys only — no session cookies from the agent

## Open Questions (Deferred to Phase 2+)

1. **Diff alerting** — Compare pslist snapshots between run N and N-1. Surface
   new/gone processes. Email/webhook on CRITICAL findings or diff delta. Phase 2.

2. **Long-poll at scale** — Fastify holds open connections per polling agent. Fine at
   small scale. When scaling: evaluate SSE or nginx upstream keepalive.
   Timeout chain (must be validated in staging):
   - Agent poll window: 30s
   - Fastify `server.keepAliveTimeout`: 36,000ms (36s) — must exceed poll window
   - `fly.toml` `[http_service] idle_timeout`: 35s — client-facing edge timeout
   - Fly.io internal (edge → Fastify) timeout: confirmed at 60s default, no config needed
   Chain: 30s poll < 36s Fastify keepalive < 60s Fly internal. Agent gets clean 204 on timeout, not a TCP reset.

3. **Dump retention + cost** — Default: delete dump after analysis, retain findings
   indefinitely. Optional 7-day retention for reinvestigation. At S3 Standard pricing
   (~$0.023/GB/month): 32GB nightly × 30 days = ~$22/server/month if retained.
